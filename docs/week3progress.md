# Week 3 Progress Report

For Week 3, I extended the AI rollout prototype by adding asynchronous prediction processing, canary rollout, operational monitoring, and a basic load testing script. In Week 2, the system supported model versioning and shadow deployment, where model v2 could run in the background without affecting the actual prediction. In Week 3, I moved the system closer to an enterprise-style AI scaling setup by allowing controlled traffic routing to model v2 and supporting queue-based async prediction requests.

I added an asynchronous prediction flow where a request can be submitted to the API and immediately receive a job ID. The prediction is then processed in the background by a worker, and the user can retrieve the result later using the job status endpoint. This simulates how enterprise AI services can handle workloads that do not require an immediate response and can benefit from queue-based processing when request volume increases.

I also added canary rollout functionality. The rollout configuration now supports enabling canary traffic, selecting a canary model, and setting a canary percentage. This allows a small percentage of real requests to be routed to model v2 while model v1 remains the main active model. I tested this by setting the canary percentage to 100 percent to confirm that all traffic was routed to model v2, and then used a 10 percent canary configuration for the load test.

In addition, I added a monitoring metrics store and a metrics endpoint. The system now tracks total requests, synchronous requests, asynchronous requests, failed requests, model v1 requests, model v2 requests, average latency, p95 latency, and queue size. These metrics help evaluate the operational behavior of the AI API beyond just model output.

I also created a basic load testing script to send both synchronous and asynchronous requests to the API. The load test executed 50 synchronous requests and 50 asynchronous requests. The system successfully processed 100 total requests with 0 failed requests. The metrics showed that 92 requests were handled by model v1 and 8 requests were handled by model v2, confirming that canary routing was working under the 10 percent rollout configuration. The queue size returned to 0 after processing, confirming that the async worker completed the submitted jobs.

This Week 3 work extends the project from shadow deployment into a more complete AI scaling and API integration testbed. The project now demonstrates synchronous prediction, asynchronous queue-based prediction, canary rollout, model usage tracking, operational metrics, and load testing. These features provide the foundation for future work such as A/B testing, automated rollback, more detailed performance benchmarking, and replacing the in-memory queue with a production-grade system such as Kafka, Redis, or Celery.
